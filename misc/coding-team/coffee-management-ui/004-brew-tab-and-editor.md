# Task 004: Quick Brew — Brew Tab + Editor Screen (Phase 4a)

## Context

The app has 3 tabs: home, add, history. We're adding a 4th "Brew" tab with the app's core screen — a brew editor where the user sets up their next brew experiment.

This is the most important screen in the app. The user wants to spend ~10 seconds setting up a brew: pick coffee, glance at stages, tweak 1-2 things, hit save. No sub-screens, no heavy forms.

Key existing code:
- `BrewFormData` already has all fields + `init(brewMethod:)` for defaults + `init(cloning:)` for cloning
- `StageFormData` + `StageType` already exist for stage state
- `BrewMethod` enum has `.v60` and `.oreaV4` with default values
- `Brew+Extensions` has convenience init and computed properties
- No BrewViewModel exists yet — you'll create one
- `Main.swift` has `Tab` enum and `TabView` — you'll add `.brew`
- `NavigationCoordinator` manages tab state and navigation paths

## Objective

Add a "Brew" tab and build the Brew Editor screen — the one-screen brew setup with inline stage editing, compact params, coffee picker, and save.

### 3. `ViewModels/BrewEditorViewModel.swift` (NEW)
`@MainActor class BrewEditorViewModel: ObservableObject`

**State:**
- `@Published var formData: BrewFormData` — initialized via `BrewFormData(brewMethod: .v60)` with default stages
- `@Published var selectedCoffee: Coffee?`
- `@Published var showValidationAlert = false`
- `@Published var validationMessage = ""`
- `@Published var isSaving = false`

**Init:**
- Takes `viewContext: NSManagedObjectContext`
- Pre-populate `formData` with V60 defaults including default stages:
  - Stage 1: Fast, 60ml, orderIndex 0
  - Stage 2: Slow, 210ml, orderIndex 1
- Set `formData.brewMethod = .v60`, `formData.grindSize = 26`, `formData.grams = 18`, `formData.temperature = 93`

**Stage management methods:**
- `addStage()` — appends a new stage (default: Slow, 60ml) at the end, auto-incrementing orderIndex
- `deleteStage(at offsets: IndexSet)` — removes stages, reindexes remaining
- `toggleStageType(at index: Int)` — toggles between .fast and .slow
- `updateStageWater(at index: Int, amount: Int16)` — updates water amount for a stage

**Computed:**
- `totalWater: Int16` — sum of all `formData.stages` water amounts
- `autoGeneratedName: String` — `"{coffee.displayName} — {formatted date}"` or `"Brew — {formatted date}"` if no coffee selected

**Save:**
- `validateAndSave() -> Bool`
- Validation: coffee must be selected (required)
- Creates `Brew` entity in context, sets all fields from formData
- Sets `brew.coffee = selectedCoffee`
- Sets `brew.name` — use formData.name if non-empty, otherwise autoGeneratedName
- Sets `brew.date = Date()`
- Sets `brew.waterAmount = totalWater`
- Creates `Stage` entities for each stage in formData.stages
- Saves context
- Posts `.brewSaved` notification
- Add `Notification.Name.brewSaved` extension

### 4. `Views/Brews/BrewEditor.swift` (NEW)
The main editor view. Use `ScrollView` (NOT `List`) for the layout — we need custom inline editing that doesn't play well with List's built-in behaviors.

Layout (top to bottom in a ScrollView):

**A. Name field**
- Text field showing `autoGeneratedName` as placeholder, editable `formData.name`
- Style: large-ish font, subdued when showing auto-generated placeholder

**B. Coffee picker**
- `@FetchRequest` for all Coffee entities sorted by name
- Picker/dropdown showing coffee names. "Select Coffee" when none chosen.
- When coffee is selected, update `viewModel.selectedCoffee`
- Style: match the picker pattern from CoffeeForm (the Roaster/Country pickers)

**C. Stages section**
- Section header: "Pour Pattern" or "Stages" with the total water shown (e.g., "Stages · 270ml")
- Each stage rendered as a `StageRowView` (see below)
- `+ Add Pour` button at bottom of stage list
- The stage list should NOT use SwiftUI `List` — use a `VStack` or `LazyVStack` inside the ScrollView with manual dividers

**D. Compact params row**
- Horizontal layout showing: Grind | Dose | Temp | Brew Method
- Each param is a tappable chip/cell that reveals inline editing
- For Grind/Dose/Temp: tap → the value becomes a text field with numeric keyboard. Tap elsewhere or hit return to confirm.
- For Brew Method: simple toggle or picker between V60 and Orea V4
- Style: compact, secondary visual weight (these change rarely). Think pill-shaped or rounded-rect cells.

**E. Water total**
- Read-only display: "Total Water: {totalWater}ml"
- Updates automatically as stages change

**F. Notes**
- Optional text field/editor, collapsed or minimal by default
- Style: subdued, at the bottom

**G. Save & Brew button**
- Prominent button at the bottom (use `StandardButton` with `.primary` style like AddCoffee/AddRoaster)
- Calls `viewModel.validateAndSave()`, dismisses/navigates on success
- Disabled state if coffee not selected (visual hint)

### 5. `Views/Components/StageRowView.swift` (NEW)
Reusable component for a single stage row. Put in Views/Components/ per CLAUDE.md.

```swift
struct StageRowView: View {
    let stage: StageFormData
    let onToggleType: () -> Void
    let onUpdateWater: (Int16) -> Void
    let onDelete: () -> Void
}
```

Layout:
- Left: Type label ("Fast" / "Slow") — tappable, calls `onToggleType`. Style: pill/badge, different color for fast vs slow (e.g., fast = caramel accent, slow = muted)
- Right: Water amount ("60ml") — tappable, becomes inline text field for editing. On commit calls `onUpdateWater`.
- Swipe left to delete (use `.swipeActions` on the row, or a custom swipe gesture if not using List. Since we're in a VStack, implement delete via the onDelete callback + a visible delete affordance OR use `.contextMenu` with delete option as a simpler alternative to custom swipe gestures in a non-List context).

**Important:** Since we're NOT in a SwiftUI List, `.swipeActions` won't work. Options for delete:
- Context menu with "Delete" option (simplest)
- Trailing delete button that appears on tap/long-press
- Just use the `onDelete` callback from a minus button visible on the row

Pick the simplest approach that works — a small trash icon or minus button on the trailing edge is fine for 4a. We can add swipe gestures later.
